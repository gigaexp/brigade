---
description: Onboard brigade to an existing project — scan codebase and write ARCHITECTURE.md
---

You are onboarding brigade to an existing project. Scan the codebase and produce
`.planning/ARCHITECTURE.md` — a machine-readable project map that `manager-agent` reads
when planning sprints. This is the brownfield counterpart to `/brigade:init` — init sets
up roles, onboard tells brigade what already exists.

## When to use

- First time setting up brigade on an existing codebase (after `/brigade:init`)
- After major structural changes (new services, moved directories, new tech stack)
- When `manager-agent` makes bad task decompositions because it doesn't understand the project

## Step 1 — Scan project structure

Read the project root. Identify:

1. **Top-level layout** — what lives where (`src/`, `apps/`, `packages/`, `server/`, `client/`, etc.)
2. **Package manager and monorepo tool** — `package.json`, `pnpm-workspace.yaml`, `Cargo.toml` workspace, `go.work`, `turbo.json`, `nx.json`
3. **Entry points** — main files, server starts, CLI entry
4. **Config files** — `tsconfig.json`, `.env.example`, `docker-compose.yml`, CI configs

Use `ls`, `Glob`, and `Read` on key files. Do not read every file — scan structure, read configs and entry points.

## Step 2 — Identify modules and their tech stacks

For each logical module/service/package:

- **Name and path**
- **Tech stack** — language, framework, key libraries
- **Role** — which dev agent would own this (frontend-agent, nodejs-agent, rust-agent, golang-agent, devops-agent)
- **Key files** — entry point, main config, test config

Read `.planning/tasks/ROLES.md` if it exists — cross-reference the roles the user already picked
with what you find in the codebase.

## Step 3 — Map dependencies between modules

Identify how modules talk to each other:

- Shared types/interfaces
- API boundaries (REST, gRPC, message queues)
- Shared packages in monorepo
- Database schemas shared across services

This is critical for `manager-agent` — it needs to know which tasks can run in parallel
(different modules) vs which have dependencies (shared interfaces).

## Step 4 — Detect test infrastructure

For each module:
- Test framework (`jest`, `vitest`, `pytest`, `cargo test`, `go test`)
- Test command (`npm test`, `cargo test`, etc.)
- Test location (`__tests__/`, `*.test.ts`, `tests/`)
- Coverage tool if configured

## Step 5 — Write .planning/ARCHITECTURE.md

Create `docs/` if missing. Write the file in this format:

```markdown
# Architecture

> Auto-generated by `/brigade:onboard` on {date}. Re-run after major structural changes.

## Project type

{monorepo | single-app | multi-service | library}

## Modules

### {module-name}
- **Path:** `{path}/`
- **Stack:** {language}, {framework}, {key libs}
- **Role:** {agent-name}
- **Entry:** `{entry-file}`
- **Tests:** `{test-command}` ({framework})
- **Key files:** {list of important files}

### {next module}
...

## Dependencies

{module-a} → {module-b}: {how they communicate}
{module-a} → {module-c}: {shared types in path/to/shared/}

## Shared resources

- **Database:** {type, schema location}
- **API contracts:** {OpenAPI spec, proto files, etc.}
- **Shared types:** {path}
- **CI/CD:** {tool, config location}

## Build & run

| Command | What it does |
|---|---|
| `{command}` | {description} |

## Notes for planning

{Anything the planner should know: areas of high coupling, files that change together,
known pain points, performance-sensitive paths}
```

## Step 6 — Commit

```bash
git add .planning/ARCHITECTURE.md
git commit -m "docs: map codebase architecture"
```

## Step 7 — Report

Print:
1. Number of modules found
2. Roles matched (from ROLES.md) vs unmatched modules
3. Suggest `/brigade:plan <goal>` as next step

## Hard rules

- **Read-only scan.** Do not modify any source code. Only create `.planning/ARCHITECTURE.md`.
- **Do not guess stacks.** Read `package.json`, `Cargo.toml`, `go.mod`, etc. If unclear, say "unknown".
- **Do not over-detail.** This is a map, not documentation. One module = 5-8 lines max.
- **Keep it machine-friendly.** `manager-agent` will parse this. Consistent format matters more than prose.
